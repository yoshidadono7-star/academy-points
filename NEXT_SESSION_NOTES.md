### どの画面の話か（要確認）

以下のどれかを指している可能性が高い:
- **A. 管理画面 `/` (index.html)** — 塾長がログインした時のダッシュボード。今は「HERO'S 管理画面」ヘッダー + ダッシュボード/日常業務/学習分析/ゲーム運営/マスター設定の 2 段タブ構造。
- **B. 教室モニター `/briefing` (briefing.html)** — 教室の大型モニターに常時表示する画面。無認証でアクセス可。「ブリーフィング」という呼称。
- **C. 生徒画面 `/student`** — 宇宙テーマで「宇宙船は静かに待機している」等のミッション的表現がある。

**直前の会話の流れから、恐らく A（管理画面のダッシュボード）** を「ミッションコントロール」と呼んでいた可能性が高い。要確認。

### 要求内容
- 画面の **上半分 or 上何分の 1** に領域を取る
- **キャラクターが表示**される（顔アイコン + 吹き出し等）
- キャラクターが「**その日のメイン**」を語る
  - 例: 「今日は期末テスト対策週間の 3 日目だよ」
  - 例: 「今日の目標: 新入生面談 2 件、保護者連絡 3 件」
  - 例: 「今日は花子さんの誕生日だよ、おめでとう！」
- **質問・やりとりは不要**。一方的にキャラクターが喋るだけの情報表示。
- 見た目は BuddySystem と同じか、新規キャラクターか？

### 実装案のアイデア（参考）

**案 A: AI 自動生成型**
- index.html / briefing.html の上部に固定領域を追加
- 起動時に Claude API で「今日のメインメッセージ」を生成
  - prompt: 今日の授業数 / 未承認の家庭学習 / 生徒の誕生日 / 最近のレイドボス状況 / 天気（任意）

- キャラ画像 + 吹き出しで表示
- 1 日 1 回生成、Firestore にキャッシュ（コスト節約）

**案 B: 手動入力型**

**案 C: ハイブリッド**
- 基本は AI 自動生成
- 塾長が上書きできる入力欄付き

## 🔒 要望 2: オーナー画面のアクセス制御を強化したい（要検討）

### 現状の問題

- `makeMeOwner()` は開発用のバックドア。Console が使える人なら誰でも実行可能
- `admin@heros.jp` の Firebase Auth アカウントにログインできれば、誰でもオーナーに昇格できる
- スタッフ資料 `STAFF_GUIDE_Feature9.pdf` には `makeMeOwner()` の存在が記載されている

### 次回セッションで対応したい

1. **`makeMeOwner()` を本番から削除**（or 開発環境フラグで無効化）
   - 昇格は Firestore Console から手動で `users/{uid}.role = 'owner'` に設定
2. **スタッフ用資料と塾長用資料を分離**
   - スタッフ向け: オーナー画面の章（第 7 章）を削除した版
   - 塾長向け: 現在の完全版
3. **admin@heros.jp のパスワード変更**を検討
   - 現在 `0070heros` は共通知。スタッフが知ってる可能性
4. **Firestore Rules の強化**
   - `users/{uid}` の role 書き換えを厳しく制限（owner 同士の承認制など）

### スタッフへの資料共有時の注意（今すぐ）

**現状の `STAFF_GUIDE_Feature9.pdf` をそのままスタッフに渡すと、第 7 章で `makeMeOwner()` の存在を

→ 暫定対応: スタッフ共有前に「第 7 章を除いた版」を次回セッションで作る。または今回作成した版は塾長の
